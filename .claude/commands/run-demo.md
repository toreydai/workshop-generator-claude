# /run-demo — 执行单个 Demo 并更新文档

读取指定编号的 Demo 文档，在后台 Agent 中执行，记录结果，自动修正文档错误，并在完成后同步元数据文件。

## 输入格式

$ARGUMENTS 示例：
- `01` — 执行 demo01（含清理）
- `03 --keep` — 执行 demo03，跳过清理步骤，保留实验资源
- `01 --dry` — 只读取和展示，不实际执行命令

## 执行步骤

### Step 1：读取文件

在主线程中完成：

1. 读取当前目录的 `CLAUDE.md`，获取环境约束
2. 在 `docs/` 目录查找编号匹配的文件：
   ```bash
   ls docs/demo$ARGUMENTS*.md 2>/dev/null || ls docs/demo0$ARGUMENTS*.md 2>/dev/null
   ```
   若找不到文件，报错并停止。
3. 读取该 Demo 文件的完整内容

### Step 2：启动后台 Agent

使用 Agent 工具，设置 `run_in_background: true`，将以下内容作为 prompt 传入（替换其中的占位符）：

---
**Agent Prompt 模板：**

你是 AWS Workshop 执行助手，在 `[工作目录绝对路径]` 下运行。

执行以下 Demo 文档的全部步骤，文档内容如下：

```
[Demo 文件完整内容]
```

**执行规则：**
- 全程自主执行，不询问用户确认
- 若传入 `--dry`：展示 Demo 的标题、步骤列表和验证检查点，不执行任何命令，展示完毕后退出
- 以文档中的编号步骤为粒度上报进度，格式：`▶ 步骤N：[名称] ...` → `✅ 步骤N：[名称]` 或 `❌ 步骤N：[名称] — <错误摘要>`
- 不打印命令的原始输出（JSON、ARN、日志等），仅用于内部判断
- 文档中的"一次性状态检查"步骤如返回未就绪状态，轮询等待至完成（最长 5 分钟）再继续
- 清理步骤：若未传 `--keep` 则必须执行；传入 `--keep` 则跳过清理，并在执行摘要中注明"资源已保留"

**验证检查点执行规则：**
- 文档 `## 验证检查点` 中每条检查必须逐条执行
- 将命令实际输出与"期望精确输出"做**字符串精确匹配**（trim 后比较），不靠主观判断
- 匹配成功 → 继续；不匹配 → 进入修复流程

**修复与重试规则：**
- 某步骤或验证检查点失败时，分析根因并尝试修复，然后**从该步骤重新执行**
- 重新执行某步骤前，若该步骤会创建已存在的资源（"already exists" 类错误），先执行对应的删除/清理操作再重试，确保幂等
- 同一步骤最多重试 **3 次**（含首次执行）
- 第 3 次仍失败：
  1. 立即执行清理步骤（尽力清理，清理失败不阻塞）
  2. 在执行记录中写入 `❌ 失败` 状态，记录失败步骤、错误内容、已尝试的修复方案
  3. 输出人工介入提示：
     ```
     ⛔ Demo[N] 执行中止
     失败步骤：[步骤名]
     错误摘要：[脱敏后描述]
     已尝试修复：[修复方案列表]
     需要人工介入后运行 /run-demo [N] 重新执行
     ```
  4. 停止，不再继续

**输出脱敏（所有面向用户的文字均须处理）：**
- AWS Account ID（12 位数字）→ `<ACCOUNT_ID>`
- ARN 中的账号段同样替换
- `assumed-role/<RoleName>/<SessionName>` → `assumed-role/<ROLE>`
- EC2 实例 ID（`i-` + 17 位十六进制）→ `<INSTANCE_ID>`
- 桶名含账号数字时整体 → `<BUCKET_NAME>`
- 错误信息脱敏后再展示，不透传原始报错

**完成后执行以下操作：**

**A. 写执行记录**

在 `execution-records/execution-log.md` 中**覆盖**该 Demo 的记录（全部脱敏）：
- 若文件中已存在 `## Demo[N]` 条目，用新结果替换整个条目
- 若不存在，追加到文件末尾

格式：
```markdown
## Demo[N] — [标题]

- **最近执行**：[YYYY-MM-DD HH:MM]
- **状态**：✅ 完成 / ❌ 失败 / ⚠️ 有偏差
- **执行次数**：[N]（若大于 1，括号内注明"前 N-1 次失败"）
- **耗时**：[X] 分钟（本次）
- **失败步骤**：[步骤名，仅失败时填写]
- **偏差记录**：[详情或"无"]
- **文档更新**：[已修正项目或建议]
- **Workshop 元数据同步**：[README.md/CLAUDE.md 的更新情况，无变化则填"无"]
```

执行次数的计算：读取已有条目中的执行次数字段，加 1；若无已有记录则为 1。

**B. 修正 docs/ 文档**

若执行中发现文档问题（命令错误、参数废弃、缺少前置步骤、预期输出有误、清理不完整），直接修改 `docs/` 下对应文件并在执行记录中注明"已自动更新"；设计层面的问题只记录建议不修改。

**禁止修改以下结构性章节**（仅允许修正其中明显与执行结果不符的事实性错误）：`## 实验简介`、`## 验收标准`、`## 实验总结`。这些章节由 gen-demo 维护，不属于执行修正范围。

**C. 同步 Workshop 元数据**

每次 Demo 执行完成后，统一同步 Workshop 级别的元数据文件：

**C1. 同步 README.md**

扫描 `docs/` 下所有 `demo*.md` 文件，提取每个文件的一级标题（`# Demo0N — 标题`）。
找到 README.md 中的 Demo 列表区域，用当前 docs/ 实际文件重建该区域的链接列表：

```markdown
- [Demo01 — 实际标题](docs/demo01-xxx.md)
- [Demo02 — 实际标题](docs/demo02-xxx.md)
```

只更新列表部分，保留 README.md 的其他章节不变。若 Demo 标题未变、文件未增减，则跳过。

**C2. 同步 CLAUDE.md**

> 仅操作 `CLAUDE.md`，不得修改 `kiro-cli-system-prompt.md`。

主动回顾本次执行过程，逐一检查以下内容：
- 执行中遇到的所有报错（含已修复的）
- 对 demo 文档做过的每处修改（命令修正、参数更新、预期输出调整）
- 文档中 `> ⚠️` 注意事项在实际执行时是否得到验证或补充
- 验证检查点中发现的 API 行为与文档描述的偏差

将上述发现与 CLAUDE.md 现有 `## 已知问题` 逐条比对，把**未收录的条目**追加：
```
- **[服务/操作]**：[问题描述及建议做法]
```
若章节不存在则新增。保持精简风格。经过主动回顾确认无新内容，才可跳过。

**D. 输出执行摘要**（不含任何敏感信息）：
```
Demo[N] — [标题]
状态：✅ 完成 / ❌ 失败（需人工介入）
耗时：X 分钟
重试：N 次
偏差：[条数] 条（已修正 N 条）
Workshop 元数据同步：README.md [已更新/无变化] | CLAUDE.md [已更新/无变化]
```

---

### Step 3：通知用户

Agent 启动后，主线程立即向用户输出：

```
Demo[N] — [标题] 已在后台启动，完成后会通知你。
```
