# /run-all — 串行执行所有 Demo

扫描 `docs/` 目录，在后台 Agent 中按编号顺序依次执行所有 Demo，最终输出汇总报告并同步元数据文件。

## 输入格式

$ARGUMENTS 示例：
- （空）— 从 demo01 跑到最后一个（含清理）
- `3-4` — 只跑 demo03 和 demo04
- `2-` — 从 demo02 跑到最后
- `--keep` — 执行全部 demo，跳过每个 demo 的清理步骤，保留实验资源
- `--dry` — 只读取并展示所有 demo 内容，不实际执行任何命令

## 执行步骤

### Step 1：扫描并过滤 Demo 列表

在主线程中完成：

```bash
ls docs/demo*.md | sort
```

根据 $ARGUMENTS 过滤范围：
- 空参数：全部执行
- `N-M`：只执行编号 N 到 M
- `N-`：从编号 N 执行到最后

读取 CLAUDE.md 和所有目标 Demo 文件的完整内容。

### Step 2：启动后台 Agent

使用 Agent 工具，设置 `run_in_background: true`，将以下内容作为 prompt 传入（替换占位符）：

---
**Agent Prompt 模板：**

你是 AWS Workshop 执行助手，在 `[工作目录绝对路径]` 下运行。

依次执行以下 [N] 个 Demo，每个完成后再开始下一个：

[Demo01 文件名]：
```
[Demo01 完整内容]
```

[Demo02 文件名]：
```
[Demo02 完整内容]
```

（以此类推）

**执行规则：**
- 全程自主执行，不询问用户确认
- 若传入 `--dry`：逐个展示每个 Demo 的标题、步骤列表和验证检查点，不执行任何命令，展示完毕后退出
- 每个 Demo 内以编号步骤为粒度上报进度：`▶ 步骤N：[名称] ...` → `✅ / ❌`
- 每个 Demo 完成后打印单行进度：`[X/N] DemoXX ✅ 完成 (X分钟)` 或 `[X/N] DemoXX ❌ 失败（需人工介入）`
- 不打印命令的原始输出（JSON、ARN、日志等），仅用于内部判断
- 文档中的"一次性状态检查"步骤如返回未就绪，轮询等待至完成（最长 5 分钟）再继续
- 清理步骤：若未传 `--keep` 则每个 Demo 完成后必须执行；传入 `--keep` 则全程跳过清理，在最终摘要中注明"所有资源已保留"

**验证检查点执行规则：**
- 每个 Demo 的 `## 验证检查点` 中每条检查必须逐条执行
- 将命令实际输出与"期望精确输出"做**字符串精确匹配**（trim 后比较），不靠主观判断
- 匹配成功 → 继续；不匹配 → 进入修复流程

**修复与重试规则：**
- 某步骤或验证检查点失败时，分析根因并尝试修复，然后从该步骤重新执行
- 重新执行某步骤前，若该步骤会创建已存在的资源（"already exists" 类错误），先执行对应的删除/清理操作再重试，确保幂等
- 同一步骤最多重试 **3 次**（含首次执行）
- 第 3 次仍失败：
  1. 执行当前 Demo 的清理步骤（尽力清理，清理失败不阻塞）
  2. 在执行记录中写入 `❌ 失败` 状态，记录失败步骤、错误内容、已尝试的修复方案
  3. 停止整个流程，输出人工介入提示：
     ```
     ⛔ Demo[N] 执行中止，后续 Demo 已跳过
     失败步骤：[步骤名]
     错误摘要：[脱敏后描述]
     已尝试修复：[修复方案列表]
     需要人工介入后运行 /run-demo [N] 单独重试，再运行 /run-all [N+1]- 继续
     ```
  4. 不自动跳过继续执行后续 Demo

**输出脱敏（所有面向用户的文字均须处理）：**
- AWS Account ID（12 位数字）→ `<ACCOUNT_ID>`
- ARN 中的账号段同样替换
- `assumed-role/<RoleName>/<SessionName>` → `assumed-role/<ROLE>`
- EC2 实例 ID（`i-` + 17 位十六进制）→ `<INSTANCE_ID>`
- 桶名含账号数字时整体 → `<BUCKET_NAME>`
- 错误信息脱敏后再展示，不透传原始报错

**每个 Demo 完成后：**

在 `execution-records/execution-log.md` 中**覆盖**该 Demo 的记录（脱敏），格式与 /run-demo 相同（含 `执行次数`、`元数据同步` 字段）。若已存在该 Demo 条目则替换，不存在则追加。执行次数在已有值基础上加 1。

若发现文档问题（命令错误、参数废弃、预期输出有误、清理不完整），直接修改 `docs/` 对应文件，在执行记录中注明"已自动更新"。**禁止修改结构性章节** `## 实验简介`、`## 验收标准`、`## 实验总结`（仅允许修正明显与执行结果不符的事实性错误），这些章节由 gen-demo 维护。

**全部 Demo 执行完成后，执行以下操作：**

**A. 写汇总记录**

输出汇总表并追加到 `execution-records/execution-log.md`：
```markdown
## 执行汇总

| Demo | 标题 | 状态 | 耗时 | 偏差数 |
|------|------|------|------|--------|
| 01   | ...  | ✅   | X min | 0 |
...

- **总耗时**：X 分钟
- **总偏差**：N 条（已自动修正 N 条）
- **Workshop 元数据同步**：[本次整体同步情况]
```

**B. 同步 Workshop 元数据（全量扫描）**

所有 Demo 执行完成后，统一对 Workshop 级别的元数据文件做一次全量同步：

**B1. 同步 README.md**

扫描 `docs/` 下所有 `demo*.md` 文件，提取每个文件的一级标题，按编号排序，完整重建 README.md 中的 Demo 列表区域。只更新列表区域，保留其他章节不变。

**B2. 同步 CLAUDE.md**

> 仅操作 `CLAUDE.md`，不得修改 `kiro-cli-system-prompt.md`。

主动回顾所有 Demo 的执行过程，逐一检查以下内容：
- 每个 Demo 中遇到的所有报错（含已修复的）
- 对 demo 文档做过的每处修改（命令修正、参数更新、预期输出调整）
- 执行中发现的 API 行为与文档描述的偏差
- 文档中 `> ⚠️` 注意事项在实际执行中是否得到验证或补充

将上述发现汇总去重，与 CLAUDE.md 现有 `## 已知问题` 逐条比对，把**未收录的条目**追加：
```
- **[服务/操作]**：[问题描述及建议做法]
```
若章节不存在则新增。保持精简风格。经过主动回顾确认无新内容，才可跳过。

**C. 输出最终摘要**（不含任何敏感信息）：
```
/run-all 完成
执行范围：Demo[X] — Demo[Y]
总状态：全部通过 / N 个失败
总耗时：X 分钟
文档修正：N 条
Workshop 元数据同步：README.md [已更新/无变化] | CLAUDE.md [已更新/无变化]
```

---

### Step 3：通知用户

Agent 启动后，主线程立即向用户输出：

```
共 [N] 个 Demo 已在后台启动，完成后会通知你。
执行范围：Demo[X] — Demo[Y]
```
